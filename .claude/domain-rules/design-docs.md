## Boundary

- **OWNS**: `docs/design/*.md`
- **MUST NOT modify** unless changing a top-level design decision: directory structure, module boundaries, contracts between layers, naming conventions, trust-model rules.
- **MAY READ**: any source file, manifest, test, or benchmark — to verify a proposed wording against current behavior.

______________________________________________________________________

## Core Rule

**Design docs record top-level, universally-applicable design decisions. They do not record implementation details.**

A design doc is the contract a reader can rely on when there is no code to read. Implementation details are read from the code; documenting them in prose creates two sources of truth that drift.

______________________________________________________________________

## What belongs in design docs

- Architectural decisions (directory layout, module responsibilities, layering rules).
- Contracts between layers (what each layer owns, must not do, may read).
- Naming conventions and structural invariants (e.g. "manifest keys are PascalCase").
- Trust-model boundaries and the *reasons* behind them.
- The *why* behind non-obvious decisions, when future readers would not derive it from the code.

## What does NOT belong in design docs

- Enumerations of current files, classes, or ops (these change with implementation; readers should `ls` or `grep`).
- Specific function names, parameter lists, or call signatures (these belong in docstrings or reference docs autogenerated from code).
- Step-by-step procedures that mirror what the code already does.
- Counts, line numbers, line-count budgets for specific files, or other quantitative snapshots that decay the moment the code changes.
- Migration history, "we used to do X but now do Y" — that lives in commit messages and PR descriptions.

______________________________________________________________________

## When a reviewer asks for a doc update

**Default answer is no.** A reviewer pointing at a design doc and saying "this is now stale" almost always means one of:

1. **The implementation legitimately changed a design decision** — update the doc with the *new decision*, not the new implementation. Wording should still be at the design layer ("families may be sharded across multiple files"), not the implementation layer ("currently sharded into A.yaml, B.yaml, C.yaml").
1. **The implementation added detail that does not violate the existing design decision** — do **not** update the doc. The doc is still correct at the design layer; pushing implementation specifics into it would degrade the doc, not improve it. Reply on the review thread with that rationale and reject the change.
1. **The doc has an actual factual error or contradicts current behavior at the design layer** — fix it, but at the design layer.

If you cannot tell which case applies: the test is "could this wording survive the next 5 implementations of this thing?" If yes, it's design-layer. If no, it's an implementation snapshot — leave it out.

## Examples

| Bad (implementation snapshot)                                                                                                                                           | Good (design decision)                                                                                                                                                |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| "The elementwise family is sharded into `elementwise_unary_math.yaml`, `elementwise_unary_activation.yaml`, `elementwise_binary.yaml`, `elementwise_multi_input.yaml`." | "Large families may be sharded across multiple files."                                                                                                                |
| "Validator currently checks 5 levels: L0 schema, L1 signature, L2 shape, L3 dtype, L4 benchmark."                                                                       | "Validator enforces manifest ↔ code consistency in layered checks; see [scripts/validate_manifest.py](../../scripts/validate_manifest.py) for the level definitions." |
| "The `Op` base class has 7 attributes: `_static_axes`, `_kernel_class`, ..."                                                                                            | "Each subclass declares the static axes used for cache keys; see the base class for the attribute names."                                                             |
| "After PR #1094 the loader uses `yaml.safe_load(text) or {}` directly."                                                                                                 | (delete — this is changelog material, not design)                                                                                                                     |
